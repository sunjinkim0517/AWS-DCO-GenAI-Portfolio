## 2. 내가 만든 DCO 인프라 용어 정의

### Rack
내 정의: 서버, 스위치 같은 장비를 세로로 층층이 꽂아 넣는 철제 선반입니다. 아파트처럼 한 칸(랙 유닛, U)마다 장비를 끼워 넣는다고 생각하면 됩니다.

### Server
내 정의: 실제로 연산을 하고 데이터를 처리하는 컴퓨터 본체입니다. 우리가 쓰는 PC와 원리는 같지만, 랙에 들어가도록 납작하고 길쭉한 모양입니다.

### NIC
내 정의: 서버가 네트워크 케이블을 꽂아 다른 장비와 통신할 수 있게 해주는 부품입니다(Network Interface Card). 서버의 "귀와 입" 역할을 하는 포트라고 이해했습니다.

### ToR Switch
내 정의: Top of Rack Switch의 줄임말로, 랙 맨 위에 달려서 그 랙 안 서버들을 한 데 모아 상위 네트워크로 연결해주는 스위치입니다. 랙 단위의 "중간 정류장" 같은 역할입니다.

### Link Down
내 정의: 케이블이 빠지거나 포트가 꺼지는 등, 두 장비 사이의 물리적 연결 자체가 끊긴 상태를 뜻합니다. "연결이 살아있는지 아닌지"를 나타내는 표현이라고 이해했습니다.

### CRC error
내 정의: 연결은 살아있지만, 전송된 데이터가 오는 도중 손상돼서 원래 값과 다르게 도착한 걸 감지했다는 신호입니다. 연결 여부(Link Down)와는 다르게, 데이터 자체의 오류를 알려주는 신호로 구분했습니다.

### SSD failure
내 정의: 데이터를 저장하는 SSD 저장장치가 정상적으로 동작하지 않거나 곧 고장 날 조짐을 보인다는 경고입니다.

### Fan Alert
내 정의: 장비 내부 온도를 식혀주는 냉각 팬이 느려지거나 멈췄을 때 뜨는 경고입니다. 팬이 멈추면 결국 과열로 이어질 수 있어서 초기에 확인해야 하는 신호라고 이해했습니다.

## 3. 헷갈리는 용어 비교

내가 비교한 용어: Link Down vs CRC error

내가 이해한 차이:
Link Down은 "연결 자체가 끊어졌는가"의 문제이고, CRC error는 "연결은 되어 있지만 데이터가 오는 중 손상됐는가"의 문제입니다. 비유하자면 Link Down은 전화가 아예 끊긴 상태, CRC error는 전화는 연결돼 있는데 잡음이 섞여 말이 깨져 들리는 상태에 가깝다고 이해했습니다.

아직 헷갈리는 부분: CRC error가 반복적으로 쌓이면 결국 Link Down으로 이어질 수 있는지, 아니면 서로 완전히 독립적인 신호인지는 아직 확실하지 않습니다.

## 4. 앞으로 사용할 학습 보조 프롬프트

### 프롬프트 1. 모르는 DCO 용어 설명용
역할: 데이터센터 운영 직무를 처음 배우는 학생을 돕는 학습 도우미
목표: 모르는 DCO 인프라 용어를 비전공자도 이해할 수 있게 쉽게 설명
상황: AWS DCO 인턴십 준비를 위한 교육용 학습이며, 실제 AWS 내부정보가 아닌 공개 직무 맥락 기준으로 설명 요청
출력 형식: 쉬운 설명 / 실무에서 나올 수 있는 상황 / 내가 자기 말로 바꿔 쓸 때 참고할 표현 / 주의할 점
제한 조건: 실제 장비명·티켓 번호·계정정보 생성 금지, 원인 단정 금지, SAMPLE/DEMO/EDU 접두어만 사용

### 프롬프트 2. 비슷한 용어 비교용
역할: DCO 인프라 용어를 쉽게 비교해 주는 학습 도우미
목표: 헷갈리는 두 용어의 차이를 비전공자도 이해할 수 있게 설명
상황: 두 용어가 비슷하게 느껴져서 구분 연습이 필요한 상황
출력 형식: 용어 A 쉬운 의미 / 용어 B 쉬운 의미 / 가장 큰 차이 / 교육용 예시 / 기억법 / 주의할 점
제한 조건: 실제 내부정보·실제 장비명·실제 티켓 번호 금지, 원인 단정 금지

### 프롬프트 3. 장애 표현 확인 순서 정리용
역할: 장애 표현을 보고 확인 순서를 정리해주는 학습 도우미
목표: 장애처럼 보이는 표현이 나왔을 때, 원인을 단정하지 않고 확인해야 할 순서를 단계별로 정리
입력: 교육용 샘플 로그 또는 티켓 문구 (예: "SAMPLE-RACK-01에서 Link Down 이벤트 관찰")
출력 형식: 가능한 원인 목록 / 확인해야 할 순서(단계별) / 교육용 주의 문구
제한 조건: 원인을 하나로 단정하지 않기, 실제 장비 점검 지시처럼 표현하지 않기, 실제 정보 생성 금지

## 5. AI 답변을 보고 내가 수정한 점

어려워서 쉽게 바꾼 표현:
"물리 계층 연결 상태(Physical Layer Connectivity)"처럼 딱딱한 표현을, "케이블이 꽂혀서 연결이 살아있는지"처럼 일상적인 말로 바꿔 썼습니다.

실제 정보처럼 보일 수 있어 삭제하거나 수정한 표현:
용어 설명 예시에 등장할 뻔한 구체적인 장비 모델명이나 IP 주소는 전부 빼고, SAMPLE/EDU 같은 교육용 표현만 남겼습니다.

너무 단정적이라 바꾼 표현:
"CRC error는 케이블 불량 때문이다"처럼 원인을 하나로 단정한 문장을, "케이블 문제일 수도 있고, 다른 요인일 수도 있어 확인이 필요하다"는 식으로 바꿨습니다.

아직 헷갈리는 용어: CRC error와 Link Down의 인과관계(반복되면 이어지는지 여부)

다시 공부하고 싶은 용어: ToR Switch가 상위 스위치(Spine)와 정확히 어떻게 연결되는 구조인지

## 6. 오늘의 느낀 점

오늘 처음 알게 된 용어는:
ToR Switch입니다. 네트워크 강의에서 스파인-리프 구조는 배웠지만, 랙 단위에서 그 시작점이 ToR Switch라는 걸 용어로 명확히 정리한 건 오늘이 처음이었습니다.

가장 헷갈렸던 용어는:
Link Down과 CRC error입니다. 둘 다 "네트워크 장애" 느낌이라 처음엔 같은 걸로 생각했는데, 연결 여부와 데이터 오류라는 다른 층위의 문제라는 걸 구분하게 됐습니다.

AI 답변을 사용할 때 조심해야 할 점은:
AI가 설명을 쉽게 해주는 대신, 원인을 너무 자신 있게 단정해버릴 수 있다는 점입니다. 실제 현장에서는 "확인 전까지는 가능성 중 하나"로 다뤄야 한다는 걸 다시 새겼습니다.

다음 차시에서 더 알고 싶은 내용은:
SOP(표준운영절차)가 실제로 어떤 형식과 수준으로 작성되는지, 4차시 SOP 분석에서 더 깊게 다뤄보고 싶습니다.
