# DCO SOP 분석 인증본

- 과정명: AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 및 포트폴리오 제작 과정
- 차시: 4차시
- 작성자: 김선진
- 작성일: 2026-07-07
- 제출 위치: 강의자 Notion 제출 페이지 하부 페이지
- 사용 데이터: 교육용 샘플 데이터 사용
- 주의: 이 문서는 실제 AWS 내부 SOP가 아니라 교육용 샘플 SOP를 바탕으로 작성한 학습 자료입니다.

---

## 1. 내가 읽은 교육용 샘플 SOP 제목

SAMPLE-DCO-SOP-04: 샘플 Link Down 이벤트 기록 및 전달 절차

---

## 2. SOP의 목적

내가 이해한 목적:
교육용 샘플 로그에서 Link Down 이벤트가 보였을 때, 발생 시간·관련 샘플 장비명·영향 범위·전달(Escalation) 필요 여부를 놓치지 않고 순서대로 정리하는 연습을 하기 위한 문서입니다. 즉, 실제 장비를 고치는 절차가 아니라 "무엇을, 어떤 순서로 기록하고 전달할지"를 익히는 학습용 절차입니다.

---

## 3. SOP 절차 요약

1. 샘플 로그에서 Link Down 이벤트가 표시된 시간을 확인한다.
2. 이벤트에 표시된 샘플 장비명을 확인한다.
3. 같은 장비에서 반복 이벤트가 있었는지 로그 안에서 확인한다.
4. Ticket opened / Ticket escalated 같은 표현이 함께 있는지 확인한다.
5. 영향 범위는 단정하지 말고, 로그에 나타난 관찰 내용만 정리한다.
6. 혼자 판단하기 어려우면 Escalation 필요 여부로 표시한다.
7. 확인한 내용을 Handover Memo나 Incident Report 초안에 넣을 수 있게 짧게 정리한다.

---

## 4. 주의사항

- 실제 장비에 접속하지 않습니다.
- 실제 네트워크 진단을 하지 않습니다.
- 실제 AWS 환경에 접속하지 않습니다.
- 실제 티켓 번호, 실제 장비 시리얼, 실제 IP를 사용하지 않습니다.
- AI 답변을 그대로 복사하지 않고 검토합니다.

내가 추가로 주의해야 한다고 생각한 점:
"반복 이벤트가 있는지 확인"과 "영향 범위를 단정하지 않기"는 서로 부딪힐 수 있다고 느꼈습니다. 반복이 있으면 심각해 보이니 원인을 단정하고 싶어지는데, SOP는 그래도 "관찰된 사실만" 적으라고 하니, 관찰(반복 횟수)과 판단(원인 추정)을 분리해서 쓰는 습관을 들여야겠다고 생각했습니다.

## 6. 내가 만든 SOP 체크리스트

- [ ] Link Down 이벤트가 발생한 시간을 기록했다
- [ ] 관련된 샘플 장비명(SAMPLE/DEMO/EDU 접두어)을 기록했다
- [ ] 같은 장비에서 반복된 이벤트가 있는지 확인했다
- [ ] Ticket opened / escalated 표현이 있는지 확인했다
- [ ] 영향 범위는 "관찰된 사실"로만 적고, 원인을 단정하지 않았다
- [ ] 혼자 판단하기 어려운 부분은 Escalation 필요로 표시했다
- [ ] Handover Memo/Incident Report에 넣을 수 있게 짧게 정리했다

## 7. 오늘 사용한 AI 프롬프트 요약

역할: DCO 직무 이해 수업에서 교육용 SOP를 쉽게 설명해 주는 학습 도우미

목표: 교육용 샘플 SOP를 목적·절차·주의사항·확인항목으로 나누고, 학생이 따라 읽기 쉬운 체크리스트로 변환

상황: AWS DCO 인턴십 직무 이해를 위한 학습, 실제 AWS 내부 SOP가 아닌 교육용 샘플 SOP(SAMPLE-DCO-SOP-04) 기준

입력 자료: SAMPLE-DCO-SOP-04 (샘플 Link Down 이벤트 기록 및 전달 절차) 전문

출력 형식: SOP 목적 / 절차 요약 / 주의사항 / 확인항목 / 체크리스트 변환 / 한 문장 요약

제한 조건: 실제 AWS 내부 절차처럼 단정하지 않기, 실제 장비명·티켓 번호·IP·계정정보 생성 금지, SAMPLE/DEMO/EDU 접두어만 사용, 원인을 하나로 단정하지 않기

---

## 8. AI 답변을 보고 내가 수정한 점

실제 절차처럼 보여 수정한 표현:
"이 SOP에 따라 실제 장비를 진단하세요" 같은 문장이 나올 뻔한 부분을, "교육용 샘플 로그를 확인하는 연습입니다"로 바꿨습니다.

실제 장비 점검처럼 보여 삭제한 표현:
ping이나 접속 명령어처럼 실제 네트워크 진단을 떠올리게 하는 표현은 전부 빼고, "로그에서 확인한다"는 표현으로 통일했습니다.

원인을 단정해서 바꾼 표현:
"Link Down은 케이블 문제다"처럼 원인을 하나로 단정한 문장을, "케이블 문제일 수도 있고 다른 요인일 수도 있어, 로그에 관찰된 내용만 우선 기록한다"로 바꿨습니다.

어려워서 쉽게 바꾼 표현:
"영향 범위(Impact Scope)"처럼 딱딱한 용어를 "이 문제가 어디까지 영향을 줬는지"처럼 풀어 썼습니다.

---

## 9. 오늘의 느낀 점

SOP를 읽고 새롭게 알게 된 점은:
SOP가 "무엇을 고치라"는 문서가 아니라 "무엇을 어떤 순서로 확인하고 기록하고 전달하라"는 문서라는 걸 명확히 알게 됐습니다. 지난 2차시에서 제가 강점으로 꼽았던 "꼼꼼한 기록"과 "규칙 준수"가 SOP를 따라가는 데 그대로 필요한 태도라는 것도 다시 확인했습니다.

SOP를 체크리스트로 바꾸면 좋은 점은:
긴 문장을 눈으로 계속 읽지 않아도, 하나씩 체크하면서 빠뜨린 게 없는지 바로 확인할 수 있다는 점입니다. 특히 교대 근무처럼 긴장되고 바쁜 상황에서는 문장보다 체크박스가 훨씬 실수를 줄여줄 것 같습니다.

AI 답변을 사용할 때 조심해야 할 점은:
AI가 SOP를 설명하다가 자연스럽게 "실제로 이렇게 진단하면 됩니다"처럼 실무 지시로 넘어갈 수 있다는 점입니다. 교육용 샘플이라는 전제를 계속 명시하지 않으면, 나도 모르게 실제 절차처럼 받아들일 수 있어서 매번 확인이 필요하다고 느꼈습니다.

다음 차시에서 준비해야 할 것은:
SOP-Ticket-Escalation-Handover Memo가 실제 교대 근무 상황에서 어떻게 하나의 흐름으로 이어지는지, 좀 더 구체적인 시나리오로 연습해보고 싶습니다.
